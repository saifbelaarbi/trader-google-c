Run the bot in the cloud so it survives your PC being off — what a trading VPS does, for a
fraction of the price, in the GCP project you already use.

<Info>
  **Why a VM, not a trading VPS or Cloud Run.** `ClaudeBreakout` acts once per **4h candle**,
  so the low-latency VPS pitch (QuantVPS/ForexVPS) buys you nothing. You need uptime +
  persistent local state. Cloud Run is stateless/scale-to-zero (right for the relay, wrong
  for a long-running bot with a local SQLite DB). A small Compute Engine VM is the right tool.
  Cost: an `e2-small` ≈ **$12–14/mo**.
</Info>

## Provision

<Steps>
  <Step title="Set project and enable APIs">
    ```bash
    gcloud config set project tradingbot-496815
    gcloud config set compute/zone europe-west1-b
    gcloud services enable compute.googleapis.com iap.googleapis.com
    ```
  </Step>
  <Step title="Lock SSH to IAP (no public SSH port)">
    ```bash
    gcloud compute firewall-rules create allow-iap-ssh \
      --network=default --direction=INGRESS --action=ALLOW \
      --rules=tcp:22 --source-ranges=35.235.240.0/20
    ```
  </Step>
  <Step title="Create the VM">
    ```bash
    gcloud compute instances create ftbot-vm \
      --machine-type=e2-small \
      --image-family=debian-12 --image-project=debian-cloud \
      --boot-disk-size=20GB --boot-disk-type=pd-balanced \
      --shielded-secure-boot --shielded-vtpm --shielded-integrity-monitoring
    gcloud compute ssh ftbot-vm --tunnel-through-iap
    ```
    The VM keeps an ephemeral public IP for **outbound** HTTPS to Bybit, but nothing inbound
    except IAP-tunnelled SSH.
  </Step>
</Steps>

## Install and run (on the VM)

```bash
sudo apt-get update && sudo apt-get install -y docker.io git
sudo usermod -aG docker "$USER"      # log out/in, then reconnect
git clone https://github.com/saifbelaarbi/trader-google-c.git
cd trader-google-c

nano ftbot/config.dry.json           # fill secrets (see Run on your PC › step 2)
git update-index --skip-worktree ftbot/config.dry.json   # protect them from git pull

docker run -d --name ftbot --restart unless-stopped \
  -v "$PWD/ftbot:/freqtrade/user_data" -p 127.0.0.1:8080:8080 \
  freqtradeorg/freqtrade:stable \
  trade --userdir /freqtrade/user_data \
        --config /freqtrade/user_data/config.dry.json --strategy ClaudeBreakout
```

`--restart unless-stopped` survives crashes and reboots; the dry-run DB persists on the
VM disk.

## Operate

<AccordionGroup>
  <Accordion title="Update after a push to main" icon="arrows-rotate">
    ```bash
    cd ~/trader-google-c && git pull origin main && docker restart ftbot
    ```
    Your filled config is protected by `skip-worktree`.
  </Accordion>
  <Accordion title="Migrate off your PC" icon="right-left">
    Run the bot in **exactly one place**. When the VM is healthy, `/stop` the PC bot and
    close it. Two instances on the same Telegram token cause update conflicts.
  </Accordion>
  <Accordion title="Back up and tear down" icon="floppy-disk">
    ```bash
    gcloud compute disks snapshot ftbot-vm --snapshot-names="ftbot-$(date +%Y%m%d)"
    gcloud compute instances stop ftbot-vm    # or: start / delete
    ```
  </Accordion>
  <Accordion title="Hardening for go-live" icon="shield-halved">
    When you switch to real keys: store `BYBIT_API_KEY`/`SECRET` in **Secret Manager** (not
    on disk), give the VM a least-privilege service account, consider Cloud NAT +
    `--no-address` for zero inbound surface, and IP-allow-list the Bybit key to the VM's
    egress IP. See [go-live](/go-live).
  </Accordion>
</AccordionGroup>
