All of this runs on the **PC or VM** — cloud sessions cannot run freqtrade. Use a separate
strategy class so the live `ClaudeBreakout` is never touched. Do the steps **in order**.

<Steps>
  <Step title="Longer history first">
    Free confidence across regimes (2021 bull, 2022 bear):
    ```bash
    freqtrade download-data --userdir ftbot --config ftbot/config.dry.json \
      --timeframes 4h --timerange 20210101-
    ```
  </Step>
  <Step title="Realism — the gate that matters">
    Re-run the baseline **with funding + a slippage assumption** and confirm the net profit
    factor before building anything.
    ```bash
    freqtrade backtesting --userdir ftbot --config ftbot/config.dry.json \
      --strategy ClaudeBreakout --timerange 20210101- --breakdown month
    ```
  </Step>
  <Step title="A/B the levers">
    Compare every variant against the live baseline in one run:
    ```bash
    freqtrade backtesting --userdir ftbot --config ftbot/config.dry.json \
      --strategy-list ClaudeBreakout ClaudeBreakoutRisk ClaudeBreakoutCorr \
                      ClaudeBreakoutPyramid ClaudeBreakoutChandelier ClaudeBreakoutPro \
      --timerange 20240601- --breakdown month
    ```
  </Step>
  <Step title="Walk-forward hyperopt">
    Optimise on history, then validate the chosen params **untouched** on a held-out period:
    ```bash
    freqtrade hyperopt --userdir ftbot --config ftbot/config.dry.json \
      --hyperopt-loss CalmarHyperOptLoss --strategy ClaudeBreakoutHO \
      --timerange 20210101-20241231 --epochs 300 --spaces buy sell stoploss

    freqtrade backtesting --userdir ftbot --config ftbot/config.dry.json \
      --strategy ClaudeBreakoutHO --timerange 20250101- --breakdown month
    ```
  </Step>
</Steps>

## Read the results the right way

Win rate is a red herring. Judge on:

<CardGroup cols={2}>
  <Card title="Profit factor (net)" icon="divide">
    Gross profit ÷ gross loss, after fees **and** funding.
  </Card>
  <Card title="Calmar" icon="wave-square">
    Return ÷ max drawdown — the primary metric, since the go-live gate is a drawdown gate.
  </Card>
  <Card title="Payoff ratio" icon="scale-unbalanced">
    Average winner ÷ average loser. The levers should fatten this.
  </Card>
  <Card title="Per-month / per-pair" icon="calendar">
    `--breakdown month` — confirm the edge isn't one pair or one quarter.
  </Card>
</CardGroup>

## Decision rule

<Note>
  Keep a change only if it improves **Calmar on the validation period, net of
  funding + slippage**, without breaking the drawdown gate. One change per backtest so
  attribution stays clean. **Adopt a parameter *region*, not the single best epoch** — a real
  edge is a plateau; a spike is noise.
</Note>

## Deeper checks

- **Monte-Carlo / trade-shuffle** on the winning config to budget the 95th-percentile
  drawdown — the realised 25.9% is one sample.
- **`freqtrade backtesting-analysis`** for entry/exit-reason and per-pair attribution.
- **`plot-dataframe` / `plot-profit`** for visual sanity checks.
